КОНТЕКСТ ДЛЯ ПРОЄКТУ CODEX LEARNING LAB

Мета файлу

Цей файл містить короткий бізнесовий і навчальний контекст для роботи в Codex. Його можна додати до проєкту Codex Learning, щоб Codex краще розумів, хто такий користувач, що таке ALTACO та EONYX, і для чого створюється навчальний сайт.

Головна мета проєкту — не створити ідеальний комерційний продукт, а практично познайомитися з можливостями Codex: роботою з файлами, кодом, терміналом, структурою проєкту, перевіркою результату та поясненням змін простою мовою.


1. Хто працює з Codex

Користувач: Олександр Швед.
Роль: директор / засновник.
Місто: Київ, Україна.

Олександр розвиває кілька напрямків:
- компанію ALTACO, яка працює з натуральним каменем і кварцовим агломератом;
- проєкт EONYX, який займається прикладними AI-рішеннями для бізнесу;
- власну практику впровадження AI-інструментів у реальні бізнес-процеси.

Олександр не є професійним розробником програмного забезпечення. Він вивчає Codex практично: через прості задачі, зрозумілі приклади, роботу з файлами, запуск коду і перевірку результатів. Пояснення мають бути українською мовою, без зайвого ускладнення і без надмірної технічної термінології.


2. ALTACO — короткий опис

ALTACO — українська компанія з Києва, яка займається дистрибуцією якісного італійського натурального каменю та кварцового агломерату.

Основні напрями ALTACO:
- імпорт і продаж натурального каменю з Італії;
- дистрибуція кварцового та мармурового агломерату Santa Margherita;
- представлення брендів Bagnara та Santa Margherita в Україні;
- продаж слябів каменю зі складу в Києві;
- робота з обробниками каменю, меблярами, архітекторами, дизайнерами, забудовниками та приватними проєктами;
- індивідуальні поставки каменю під конкретні архітектурні та інтер’єрні задачі.

Матеріали, з якими працює ALTACO:
- мармур;
- граніт;
- кварцит;
- травертин;
- онікс;
- кварцовий агломерат;
- мармуровий агломерат.

Партнери та бренди:
- Bagnara;
- Santa Margherita;
- Marmo Elite;
- GR Marmi;
- інші італійські та європейські постачальники натурального каменю.

ALTACO працює як B2B-дистриб’ютор і постачальник матеріалів для реального сектору. Компанія має практичний досвід у продажах, складі, логістиці, комерційних пропозиціях, роботі з клієнтами, проєктних поставках і технічних консультаціях щодо каменю.

Для майбутніх AI-проєктів ALTACO важлива як реальний бізнес-полігон: тут є справжні процеси, дані, клієнтські запити, складські залишки, прайси, фото матеріалів, логістика і документообіг.


3. EONYX — короткий опис

EONYX — AI deployment / R&D-проєкт, який створює прикладні AI-системи для B2B-компаній реального сектору.

Основний фокус EONYX:
- автоматизація бізнес-процесів;
- створення кастомних AI-агентів;
- цифровізація експертизи спеціалістів;
- перетворення операційних процесів на формалізовані workflow;
- інтеграція AI у продажі, склад, логістику, документообіг, аналітику та підтримку клієнтів.

EONYX не позиціонується як компанія, що “продає AI взагалі”. Ідея EONYX — не робити абстрактні чат-боти для всіх, а створювати контрольовані AI-системи, які працюють із реальними бізнес-даними, правами доступу, перевіркою результатів і конкретними задачами.

Ключова відмінність EONYX:
- AI впроваджується не як іграшка або демонстрація;
- AI має бути частиною робочого процесу;
- система повинна допомагати бізнесу приймати рішення, готувати документи, працювати з даними, не губити контекст і зменшувати людські помилки.

ALTACO є важливим практичним полігоном для EONYX, тому що в ALTACO можна тестувати AI-рішення на реальних бізнес-процесах: каталог матеріалів, склад, клієнтські запити, комерційні пропозиції, логістика, SEO, навчання персоналу та внутрішня аналітика.


4. Чому створюється Codex Learning Lab

Codex Learning Lab — це невеликий навчальний сайт, який має допомогти Олександру практично побачити повний цикл роботи Codex.

Це не комерційний сайт ALTACO або EONYX. Це навчальна лабораторія для першого знайомства з Codex.

Через цей сайт потрібно зрозуміти:
- як Codex оглядає папку проєкту;
- як Codex пояснює наявну структуру файлів;
- як Codex пропонує просту архітектуру сайту;
- як Codex створює або змінює файли;
- як Codex пише HTML, CSS і JavaScript;
- як Codex запускає перевірку через термінал;
- як Codex пояснює, що саме було змінено;
- як виглядає базовий цикл “завдання → зміни → перевірка → пояснення результату”.

Головний принцип: простота важливіша за ідеальність.

Не потрібно використовувати складні фреймворки, бази даних, бекенд, авторизацію або складну архітектуру, якщо для навчального прикладу достатньо HTML, CSS і JavaScript.


5. Бажаний сайт

Назва сайту: Codex Learning Lab.
Мова сайту: українська.
Стиль: простий, чистий, сучасний, без зайвого візуального шуму.
Призначення: навчальна сторінка, яка показує, що Codex може створити невеликий інтерактивний сайт і пояснити свою роботу.

Можлива структура сайту:

1. Hero-блок
   Назва: Codex Learning Lab.
   Коротке пояснення: “Навчальний мінісайт для знайомства з Codex: файли, код, термінал і перевірка результату”.

2. Блок “Що я тут вивчаю”
   Картки або список:
   - огляд проєкту;
   - створення файлів;
   - редагування коду;
   - запуск перевірки;
   - пояснення змін.

3. Блок “ALTACO як бізнес-кейс”
   Коротко пояснити, що ALTACO — реальна компанія з дистрибуції натурального каменю та кварцового агломерату, на прикладі якої можна вивчати бізнес-автоматизацію.

4. Блок “EONYX як AI-напрям”
   Коротко пояснити, що EONYX створює прикладні AI-системи для B2B-компаній і використовує реальні процеси як основу для навчання та розробки.

5. Інтерактивні елементи
   Додати 2–4 прості інтерактивні елементи, наприклад:
   - кнопка зміни теми: світла / темна;
   - чекліст “Що я вже зрозумів у Codex”;
   - простий лічильник прогресу;
   - кнопка, яка показує коротке пояснення, що таке HTML, CSS і JavaScript;
   - невеликий перемикач між контекстом ALTACO та EONYX.

6. Блок “Що має показати Codex”
   Короткий список очікуваних дій:
   - оглянути папку;
   - запропонувати структуру;
   - створити сайт;
   - додати інтерактивність;
   - перевірити результат;
   - пояснити створені файли.


6. Технічні обмеження

Не ускладнювати технології без потреби.

Бажано почати з найпростішого варіанту:
- index.html;
- styles.css;
- script.js;
- README.md.

Якщо папка вже має іншу структуру, Codex має спочатку оглянути її і запропонувати, як краще діяти.

Не потрібно одразу використовувати:
- Next.js;
- React;
- TypeScript;
- Tailwind;
- базу даних;
- авторизацію;
- серверну логіку;
- складний деплой.

Мета — зрозуміти базовий цикл роботи Codex, а не будувати production-систему.


7. Як Codex має поводитися в цьому навчальному проєкті

Codex має діяти послідовно:

1. Спочатку оглянути поточну папку проєкту.
2. Коротко пояснити, що в ній є.
3. Запропонувати просту структуру сайту.
4. Реалізувати сайт українською мовою.
5. Додати кілька інтерактивних елементів.
6. Перевірити, що сайт працює.
7. Показати, які файли створено або змінено.
8. Пояснити важливі дії простою мовою.

Пояснення мають бути дружні, короткі й практичні. Важливо не перевантажувати користувача термінами.

Приклад хорошого пояснення:
“Я створив index.html — це основна сторінка сайту. У styles.css виніс оформлення, щоб HTML не був перевантажений. У script.js додав просту інтерактивність: перемикання теми та чекліст прогресу”.

Приклад небажаного підходу:
“Я встановив складний фреймворк, налаштував багато залежностей і створив архітектуру, яка поки не потрібна для навчальної задачі”.

Хочу практично познайомитися з можливостями Codex.

Створи в цьому проєкті невеликий інтерактивний сайт “Codex Learning Lab”, через який я зможу побачити повний цикл роботи:

1. Спочатку оглянь поточну папку й коротко поясни, що в ній є.
2. Запропонуй просту структуру сайту.
3. Реалізуй сайт українською мовою.

6. Покажи, які файли ти створив і що саме змінив.

Не ускладнюй технології без потреби. Мета — не ідеальний продукт, а зрозуміти роботу Codex із файлами, кодом, терміналом і перевіркою результату.


9. Критерії готовності

Результат можна вважати успішним, якщо:
- у проєкті є простий сайт українською мовою;
- сайт можна відкрити локально в браузері;
- є зрозуміла структура файлів;
- є хоча б кілька інтерактивних елементів;
- Codex пояснив, які файли створив або змінив;
- Codex перевірив результат доступним способом;
- Олександр після виконання задачі краще розуміє, як Codex працює з кодом і файлами.
